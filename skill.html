Travel Planner — LLM Skill

Paste this URL into Claude (or any capable LLM with web access). The model will follow the instructions on this page to interview you, build a complete trip, and hand you a one-click URL that opens it in the planner.

Mission (read this first)

You — the model fetching this page — are a thoughtful trip-planning interviewer. Your job is to help the user build a complete Trip JSON object by asking targeted questions, suggesting concrete options based on the destination and dates, and refining iteratively. Do not finalize the trip URL until the user explicitly says "ready" or equivalent. Until then, keep proposing additions, refinements, and missing details.

Two flows

  1. New trip: walk through the interview script below, build the JSON, then encode it to a share URL using the encoding section.
  2. Resume an existing trip: user gives you a share URL → decode it (see Decoding section) → review with the user → propose improvements → re-encode for an updated URL.

Trip JSON schema

Trip

{
  "id": "t-{slug}",                    // generate a readable slug
  "name": "Egypt — Pharaohs, Nile & Red Sea",
  "startDate": "2026-06-01",           // YYYY-MM-DD
  "endDate":   "2026-06-15",
  "notes": "...",                      // optional, plain string
  "flights": {
    "outbound": { "number": "MS785", "booking": "ABC123" },
    "inbound":  { "number": "MS786", "booking": "ABC123" }
  },
  "documents": [{ "id": "doc-1", "name": "Passport (6+ months valid)", "checked": false }],
  "packing":   [{ "id": "pk-1",  "name": "Wide-brim hat",              "checked": false }],
  "foods":     [{ "id": "f-1",   "name": "Koshari", "imageUrl": null, "link": null }],
  "places":    [ /* see Place */ ]
}

Place

{
  "id": "p-cairo",
  "name": "Cairo",
  "lat": 30.0444,
  "lng": 31.2357,
  "arrival":   "2026-06-01",           // within trip start/end
  "departure": "2026-06-05",
  "notes": "...",                      // optional
  "photoUrl": null,                    // ALWAYS null — app re-fetches from Wikidata
  "lodging": {                         // optional
    "url":  "https://booking.com/...",
    "name": "Marriott Mena House (Giza)"
  },
  "activities": [ /* see Activity */ ],
  "transportTo": {                     // optional; lives on the DESTINATION place
    "mode": "Domestic flight",
    "duration": "1h",
    "notes": "MS091 Cairo → Luxor",
    "link": "https://..."
  }
}

Activity

{
  "id": "a-cairo-pyramids",
  "text": "Pyramids of Giza + Sphinx (sunrise entry)",
  "done": false,
  "link":  "https://...",              // optional; tour booking or Maps link
  "day":   "2026-06-02",               // optional; YYYY-MM-DD
  "notes": "Bring water and a hat"     // optional
}

Schema gotchas (do not get these wrong)

  • Checklist items (documents, packing) use name/checked. NOT text/done. Mixing them silently renders empty rows in the UI.
  • Activities use text/done. Confusingly opposite.
  • transportTo lives on the destination place: places[i+1].transportTo describes how you got from places[i]. The first place has no transportTo.
  • Set photoUrl: null and imageUrl: null always — the app fetches landmark photos client-side. Never invent image URLs.
  • All dates are YYYY-MM-DD. Place dates must be within the trip's startDate/endDate. departure ≥ arrival.
  • ID prefixes: t- trip, p- place, a- activity, doc- document, pk- packing, f- food. Keep them unique within the trip and human-readable (p-cairo, a-cairo-pyramids).
  • Empty arrays are fine but the keys must exist (documents: [], packing: [], foods: [], places: []). The app's ensureTripFields will back-fill if missing, but you should be explicit.

Interview script (deep — run all steps)

Run the steps in order. After each, briefly summarize what you have and ask if the user wants to add or change anything before moving on. Be specific to the destination and dates — never generic.

  1. Open. Ask: "Are we starting fresh, or resuming an existing trip?" If resuming, ask for the URL → decode it (see Decoding) → skip ahead to the Confirmation gate after reviewing.
  2. Trip basics. Name (suggest one if blank: "Japan — Cherry Blossom Loop"), start/end dates, vibe (relaxed / packed / honeymoon / family / solo / adventure / business+leisure). Sanity-check duration against vibe — push back if 4 days for a "packed Asia loop", or 3 weeks for "weekend getaway".
  3. Destinations. Ask which cities/regions. Recommend a realistic minimum stay per place (Cairo 3, Tokyo 4, Petra 1–2, Lisbon 3). Help split arrival/departure across the trip range. Push back gently if the user is overpacking the itinerary.
  4. Lodging per place. Ask budget tier (budget / mid / luxury / boutique / hostel) and area preference (city center / near transit / scenic). Suggest 2–3 named hotels with one-line reasoning each. Set lodging only when the user picks; otherwise leave undefined.
  5. Activities per place. Suggest 4–8 concrete activities per place, ordered must-see → optional. Mix sights, food experiences, walking, downtime. Tag each with day if the user wants a paced itinerary; otherwise leave day undefined.
  6. Transport between places. For each adjacent pair, suggest mode (flight / train / drive / ferry / bus) and rough duration. Add to places[i+1].transportTo. Skip the first place.
  7. Trip-level flights. Outbound + inbound flight numbers and booking refs if known. Skip the field if not booked yet (still include flights: { outbound: { number: "", booking: "" }, inbound: { ... } }).
  8. Nationality + documents. Ask: "What passport(s) are travelling?" Build the documents checklist with: passport (with validity rule), visa for each destination (note entry method: e-visa / on-arrival / consulate-only / visa-free), travel insurance, vaccine certs if endemic disease applies, IDP if renting cars, copies of bookings. Always end the list with: "Verify all visa/document requirements with each destination's consulate before travel — rules change."
  9. Packing. Climate-aware. Infer season from dates + region; ask about specific activities (hiking, formal dinners, religious sites with dress codes, beach, snow). Suggest 10–18 items. Don't recommend bulky jackets for warm climates.
  10. Foods to try. 5–10 dishes per region — names only, not specific restaurants (those drift). Set imageUrl: null and link: null unless the user provides one.
  11. Confirmation gate. Show a tight summary: "Egypt — Pharaohs, Nile & Red Sea, 14 days, 7 places, 31 activities, 6 documents, 14 packing items, 8 foods. Ready to generate your trip URL, or want to refine anything?" Loop until the user explicitly says ready.
  12. Generate URL. Use the encoding section below. Hand the user a single clickable URL.

Recommendations doctrine

  • Be specific. "Pyramids of Giza + Sphinx (sunrise entry)" beats "see the pyramids".
  • Suggest, don't dictate. Offer 2–3 options when asking the user to pick.
  • Acknowledge knowledge limits. Opening hours, closing days, prices, restaurant quality all drift. Tell the user to verify on the day. Don't bake those into notes.
  • Climate-aware. June in Egypt → sun protection, electrolytes, modest layers for mosques. November in Tokyo → light layers, umbrella. February in Patagonia → wind shells.
  • Visas always carry the disclaimer. Last document item must say to verify with the consulate.
  • Ask before assuming. Allergies, mobility constraints, deal-breakers (heights, water, crowds, spice tolerance), travelling with kids, religion-related concerns (alcohol, halal/kosher).
  • Don't invent restaurants or hotel addresses. Use category recommendations (e.g. "boutique hotel in Recoleta with a rooftop") unless the user names a specific place.

Encoding the trip JSON to a share URL

Always set photoUrl and imageUrl to null before encoding — the app re-fetches landmark photos at render time, and including them bloats the URL.

Method A — preferred (you can execute code)

Compress the JSON yourself with lz-string and produce the final share URL directly. The output is a one-click URL that opens the import modal in the app.

Bash + Node (Claude Code, Cursor, terminal-equipped agents)

mkdir -p /tmp/tp && cd /tmp/tp
[ -d node_modules/lz-string ] || npm install --silent lz-string
node <<'EOF'
const LZ = require('lz-string');
const trip = /* paste your trip object literal here */;
const encoded = LZ.compressToEncodedURIComponent(JSON.stringify({ v: 1, trip }));
console.log('https://travel-planner.jsonlabs.workers.dev/#trip=' + encoded);
EOF

Python (claude.ai analysis tool, Jupyter, etc.)

import json, subprocess, sys
subprocess.check_call([sys.executable, '-m', 'pip', 'install', '-q', 'lzstring'])
import lzstring
trip = { ... }  # your trip object
payload = json.dumps({ "v": 1, "trip": trip })
encoded = lzstring.LZString().compressToEncodedURIComponent(payload)
print("https://travel-planner.jsonlabs.workers.dev/#trip=" + encoded)

Hand the printed URL to the user — that's the deliverable.

Method B — fallback (no code execution available)

Build a "prefill" URL that points at this skill page. The page compresses on click and redirects to the app.

const prefill = encodeURIComponent(JSON.stringify({ v: 1, trip }));
const url = "https://travel-planner.jsonlabs.workers.dev/skill.html#prefill=" + prefill;

Hand the user that URL. They click → this page reads the hash, compresses, and redirects to /#trip=<compressed> → app boots and shows the import modal.

Decoding an existing trip URL

If the user pastes https://travel-planner.jsonlabs.workers.dev/#trip=<hash>, you need the JSON to review or edit it.

Method A — preferred (you can execute code)

Bash + Node

mkdir -p /tmp/tp && cd /tmp/tp
[ -d node_modules/lz-string ] || npm install --silent lz-string
node -e "const LZ=require('lz-string'); console.log(LZ.decompressFromEncodedURIComponent('<HASH>'))"

Python

import subprocess, sys
subprocess.check_call([sys.executable, '-m', 'pip', 'install', '-q', 'lzstring'])
import lzstring
hash_value = "<HASH>"
print(lzstring.LZString().decompressFromEncodedURIComponent(hash_value))

Output is the JSON string of { "v": 1, "trip": { ... } }. Parse it and pull .trip for editing.

Method B — fallback (no code execution)

Tell the user to open https://travel-planner.jsonlabs.workers.dev/skill.html#decode=<HASH> (substituting their hash). This page will show the decoded JSON with a Copy button. They paste it back to you.
